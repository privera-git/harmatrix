# Git Best Practices — Harmatrix

## Repository

The `harmatrix` repository contains all code, documentation, and configuration for the project.

El repositorio se encuentra en Github:

https://github.com/privera-git/harmatrix.git


## Workflow: GitHub Flow

`main` is always deployable. All work happens on short-lived feature branches.

```
main
 └── feature/add-chord-voicings   ← branch, PR, merge, delete
 └── fix/incorrect-interval-calc
 └── content/minor-scale-exercises
```

Never commit directly to `main`.

## Branch Naming

```
feature/   New functionality
fix/        Bug fix
content/   Musical content (chords, scales, exercises)
infra/     CI/CD, Vite config, tooling changes
docs/      GDD, ADRs, README changes
```

Format: `prefix/short-description-in-kebab-case`

Examples:
```bash
git switch -c feature/chord-recognition-engine
git switch -c fix/root-note-off-by-one
git switch -c content/pentatonic-scale-set
```

## Commits

**One logical change per commit.** A commit that mixes a bug fix with a refactor and a new feature is impossible to revert cleanly.

Commit message format:
```
<type>: <imperative short description>

# Optional body — explain WHY, not what. The diff shows what.
```

Types: `feat`, `fix`, `content`, `infra`, `docs`, `test`, `refactor`

Good:
```
feat: add chromatic transposition to chord voicing store
fix: handle undefined root note in interval calculator
docs: document Pinia store contract for game session
```

Bad:
```
wip
stuff
fix bug
update files
```

## Before Every Commit

```bash
npm run lint        # oxlint + eslint with auto-fix
npm run test:unit   # Vitest — all tests must pass
```

Type-check is enforced in `npm run build`, but run it when changing shared types:
```bash
npx tsc --noEmit
```

## Pull Requests

- PR = one unit of work. One branch, one PR, one merge, delete the branch.
- Every PR must pass `type-check`, `lint`, and `test:unit` before merge.
- Use feature flags (`src/config/features.ts`) when merging incomplete work early.
- Squash if the branch has noisy WIP commits; preserve history when commits are meaningful.

## Keeping a Branch Up to Date

Prefer rebase over merge to keep a linear history on feature branches:

```bash
git fetch origin
git rebase origin/main
```

If conflicts arise, resolve them per file, then:
```bash
git add <resolved-file>
git rebase --continue
```

Never use `git rebase` on `main` or any shared branch.

## Undoing Mistakes

| Situation | Command |
|-----------|---------|
| Undo last commit, keep changes staged | `git reset --soft HEAD~1` |
| Undo last commit, keep changes unstaged | `git reset HEAD~1` |
| Discard all uncommitted changes | `git restore .` |
| Remove a file from staging | `git restore --staged <file>` |
| Fix the last commit message (not yet pushed) | `git commit --amend` |
| Revert a merged commit on main | `git revert <sha>` (creates a new commit) |

Never use `git reset --hard` on pushed commits. Use `git revert` instead.

## Tags and Releases

Tag only from `main` after a stable milestone:

```bash
git tag -a v0.1.0 -m "Initial chord recognition prototype"
git push origin v0.1.0
```

## Useful Inspection Commands

```bash
git log --oneline --graph --all     # Visual branch overview
git diff origin/main...HEAD         # All changes since branching off main
git log origin/main..HEAD           # Commits not yet in main
git blame -L 10,25 src/stores/game.ts   # Who changed lines 10–25
git show <sha>                      # Full diff for a specific commit
```

## What NOT to Commit

- `.env` files or any file containing secrets or API keys
- `dist/` — generated by CI, listed in `.gitignore`
- `node_modules/`
- Editor-specific files not already in `.gitignore`

If a secret is accidentally committed, treat it as compromised immediately — rotate it, then use `git filter-repo` to purge the history.
